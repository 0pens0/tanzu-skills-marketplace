# Documentation Writing Guidelines

## Documentation Types
- **API Documentation**: Endpoints, parameters, responses, examples
- **Code Comments**: Explain "why", not "what" (code should be self-explanatory)
- **README**: Setup, usage, architecture overview
- **Architecture Docs**: System design, data flows, decisions
- **User Guides**: How to use the application

## Writing Principles
- Write for your audience (developers, users, operators)
- Use clear, concise language
- Include examples and code snippets
- Keep documentation up to date with code changes
- Use consistent formatting and structure

## Code Documentation
- Document public APIs thoroughly
- Explain complex algorithms or business logic
- Include parameter descriptions and return values
- Document exceptions that can be thrown
- Use Javadoc/JSDoc/docstring conventions

## README Structure
- Project description and purpose
- Prerequisites and installation
- Quick start guide
- Configuration options
- Usage examples
- Contributing guidelines
- License information

## Best Practices
- Start with "why" before "how"
- Use diagrams for complex concepts
- Include troubleshooting sections
- Link to related documentation
- Keep examples simple and focused
- Review documentation for clarity
- Update docs when changing code
